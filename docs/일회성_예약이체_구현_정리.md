# 일회성 예약 이체 실행 기능 — 구현 과정 정리

---

## 1. 초기 목표

`reserve_transfer_schedule/once` 패키지에 예약 이체 스케줄 관리(등록/조회/삭제)는 구현되어 있으나, **실제 이체를 실행하는 로직이 없었음**. 아래 요구사항을 충족하는 실행 기능을 구현하는 것이 목표.

- 스케줄러가 실행 대상 건(`transferDateTime <= now`)을 조회
- 이체를 실행한 뒤 스케줄 레코드를 삭제
- **분당 최대 1만 건** 처리
- 성공 시 레코드 삭제, 실패 시 남겨두고 다음 사이클에 재시도

## 3. 데드락 방지 — 비관적 락 적용

### 왜 락이 필요한가?

이체는 **잔액 읽기 → 검증 → 갱신**이 하나의 원자적 단위로 실행되어야 한다. 락 없이 동시에 같은 계좌에 접근하면 **Lost Update(갱신 손실)** 가 발생한다:

```
계좌 A 잔액: 1,000원

트랜잭션1 (A→B 500원 이체):  잔액 읽기 → 1,000원 확인 → 500 차감 → 500원 저장
트랜잭션2 (A→C 300원 이체):  잔액 읽기 → 1,000원 확인 → 300 차감 → 700원 저장
                                        ↑ T1 커밋 전에 읽음

결과: 잔액 700원 (T1의 차감이 사라짐)
기대: 잔액 200원 (1,000 - 500 - 300)
```

락을 걸면 T2가 T1의 커밋을 기다린 뒤 갱신된 잔액(500원)을 읽으므로 정합성이 보장된다.

### 문제 인식
2개의 account를 조회하는 로직이 4곳 존재하며, 락을 적용하되 동시 트랜잭션 시 데드락 위험도 방지해야 함.

#### 대상 4곳
1. `TransferService.simpleTransferNow()`
2. `TransferService.reserveOnceTransferNow()`
3. `TransferService.reserveRepeatTransferNow()`
4. `ReserveOnceTransferScheduleService.createReserveOnceTransferSchedule()`

### 왜 비관적 락을 선택했는가?

이체는 **잔액 확인 → 차감 → 증가**가 반드시 원자적으로 이루어져야 하는 작업이다. 낙관적 락(`@Version`)은 충돌 시 예외를 던지고 재시도해야 하는데, 이체 도메인에서는 다음과 같은 이유로 비관적 락이 더 적합하다:

- **충돌 빈도가 높을 수 있다**: 예약 이체 스케줄러가 동시에 수천 건을 병렬 처리하므로, 동일 계좌에 대한 동시 접근 가능성이 높음. 낙관적 락은 충돌이 잦을수록 재시도 비용이 급증함
- **재시도 로직의 복잡성**: 낙관적 락 충돌 시 단순 재시도가 아니라 잔액을 다시 읽고, 다시 검증하고, 다시 계산해야 함. 이체 금액은 동일하지만 잔액이 변경되었을 수 있으므로 잔액 부족으로 실패할 수도 있음
- **데이터 정합성 우선**: 금융 도메인에서는 처리량보다 정합성이 우선. 비관적 락은 "기다리더라도 반드시 정합성을 보장"하는 방식

### 적용한 전략: 계좌번호 오름차순 비관적 잠금

**원리**: 항상 작은 계좌번호부터 `SELECT FOR UPDATE`를 실행하면 순환 대기(circular wait)가 발생하지 않음.

#### 변경 파일들

**`AccountRepository.java`** — 비관적 잠금 쿼리 추가
```java
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT a FROM Account a WHERE a.accountNumber = :accountNumber")
Optional<Account> getAccountByAccountNumberForUpdate(String accountNumber);
```

**`AccountService.java`** — 정렬 후 잠금 메서드 추가
```java
public Account[] getTwoAccountsForUpdate(String accountNumber1, String accountNumber2) {
    boolean firstIsSmaller = accountNumber1.compareTo(accountNumber2) < 0;
    String first = firstIsSmaller ? accountNumber1 : accountNumber2;
    String second = firstIsSmaller ? accountNumber2 : accountNumber1;

    Account firstAccount = accountRepository.getAccountByAccountNumberForUpdate(first)
            .orElseThrow(() -> new IllegalArgumentException("Account not found: " + first));
    Account secondAccount = accountRepository.getAccountByAccountNumberForUpdate(second)
            .orElseThrow(() -> new IllegalArgumentException("Account not found: " + second));

    if (firstIsSmaller) {
        return new Account[]{firstAccount, secondAccount};
    } else {
        return new Account[]{secondAccount, firstAccount};
    }
}
```
- index 0 = accountNumber1에 해당하는 Account
- index 1 = accountNumber2에 해당하는 Account
- 내부적으로는 항상 작은 번호부터 잠금

**`TransferService.java`** — 3개 메서드에서 기존 개별 `getAccount()` 호출을 `getTwoAccountsForUpdate()`로 교체

**`ReserveOnceTransferScheduleService.java`** — `createReserveOnceTransferSchedule()`에서 동일하게 교체

## 4. 성능 논의 — 비관적 락으로 분당 1만 건이 가능한가?

### 결론: 매우 어렵다

#### 이유
비관적 락(`SELECT FOR UPDATE`)은 트랜잭션 커밋까지 row를 잠금:
```
LOCK(계좌 2개) → 잔액검증 → 잔액갱신 → 거래결과 저장 → 원장기록 → 스케줄 삭제 → COMMIT(잠금 해제)
```
- 동일 계좌가 여러 예약 건에 포함되면, 앞 건 커밋까지 뒤 건은 대기
- 병렬 처리를 해도 같은 계좌를 참조하는 건들은 직렬화됨
- 1건당 DB 라운드트립 5~6회 → 건당 수십 ms → 순차 처리 시 1만 건 = 수 분 이상

#### 추가 발견
기존 `updateBalanceBoth`는 **단일 JPQL UPDATE 문**으로 2개 row를 갱신:
```sql
UPDATE Account SET balance = CASE ... WHERE accountNumber IN (:sender, :receiver)
```
단일 SQL 문 내에서 DB가 일관된 순서로 row lock을 잡기 때문에, **이 패턴 자체의 데드락 위험은 이미 낮았음**.

## 5. 이중 잠금 구조 논의

### 현재 상태
- 계좌 조회 시: `SELECT FOR UPDATE` (비관적 락) — 트랜잭션 끝까지 잠금 유지
- 잔액 갱신 시: `updateBalanceBoth` (단일 bulk UPDATE) — 이미 잠긴 row에 다시 접근

### 이중 잠금의 실제 오버헤드

**같은 트랜잭션 내에서는 이미 획득한 락에 다시 접근할 때 대기하지 않는다.** DB 락은 트랜잭션 단위로 관리되므로, 동일 트랜잭션이 이미 잡고 있는 row에 UPDATE를 실행하면 바로 통과한다. 락 재획득 대기 같은 것은 없음.

따라서 이중 잠금의 실제 문제는 **성능 문제라기보다 설계상 비효율**:
- **불필요한 DB 라운드트립 1회**: `SELECT FOR UPDATE`로 이미 엔티티를 가져왔는데, 별도 bulk UPDATE를 또 보냄
- **영속성 컨텍스트 불일치**: `@Modifying` JPQL UPDATE는 영속성 컨텍스트를 우회하므로, 메모리의 엔티티와 DB 상태가 달라짐

### 개선 방향
비관적 락으로 엔티티를 가져왔으면, 엔티티를 직접 수정하고 JPA dirty checking에 맡기면 됨:
```java
senderAccount.setBalance(newSenderAccountBalance);
receiverAccount.setBalance(newReceiverAccountBalance);
// → 트랜잭션 커밋 시 JPA dirty checking이 자동으로 UPDATE 실행
```
bulk UPDATE 제거 → 잠금 1회, DB 라운드트립 절약

## 6. 3가지 대안 비교 및 선택

| 방식 | 장점 | 단점 |
|------|------|------|
| **1. 비관적 락 제거 + 기존 bulk UPDATE 유지** | 단순, 기존 구조 유지, 낮은 데드락 위험 | 동시 갱신 시 lost update 가능성 |
| **2. 낙관적 락 (`@Version`)** | 높은 동시성, 충돌 시만 재시도 | Account 엔티티에 version 필드 추가 필요 |
| **3. 비관적 락 유지 + 배치 파티셔닝** | 데드락 확실 방지 + 병렬 처리로 처리량 확보 | 구현 복잡도, 파티션 편향(skew) 가능성 |

## 7. 파티셔닝 전략 변화

### 첫 번째 안: 해시 기반 파티셔닝 (폐기)
```java
private int getPartitionKey(String sender, String receiver, int partitionCount) {
    String smaller = sender.compareTo(receiver) < 0 ? sender : receiver;
    return Math.abs(smaller.hashCode()) % partitionCount;
}
```
- 두 계좌번호 중 작은 쪽의 해시값으로 파티션 결정
- **문제**: `(A→B)`와 `(B→C)`가 다른 파티션에 배치될 수 있음 → 계좌 B에 대한 락 대기 발생

### 최종 안: Union-Find 기반 파티셔닝 (채택)
하나라도 같은 계좌를 공유하는 스케줄은 반드시 같은 파티션에 배치.

```
예시:
  Schedule 1: A → B
  Schedule 2: B → C
  Schedule 3: D → E

  union(A, B), union(B, C), union(D, E)
  → 집합1: {A, B, C}  집합2: {D, E}
  → Schedule 1, 2 → 파티션1 / Schedule 3 → 파티션2
  → 파티션 간 계좌 겹침 없음 → 락 충돌 0
```

## 8. 최종 구현 — Union-Find 파티셔닝 스케줄러

### `ReserveTransferExecutionScheduler.java`

```java
@Component
@RequiredArgsConstructor
@Slf4j
public class ReserveTransferExecutionScheduler {
    private final ReserveOnceTransferScheduleService reserveOnceTransferScheduleService;
    private final ReserveTransferExecutionService reserveTransferExecutionService;
    @Qualifier("asyncReserveTransferExecutor")
    private final Executor asyncReserveTransferExecutor;

    @Scheduled(cron = "*/30 * * * * *")
    public void executeReserveOnceTransferExecution() {
        LocalDateTime now = LocalDateTime.now();
        List<ReserveOnceTransferSchedule> schedules
                = reserveOnceTransferScheduleService.getPendingReserveOnceTransferSchedules(now);

        // 1. Union-Find로 같은 계좌를 공유하는 스케줄을 같은 집합으로 묶기
        Map<String, String> parent = new HashMap<>();
        for (ReserveOnceTransferSchedule schedule : schedules) {
            union(parent, schedule.getSenderAccountNumber(), schedule.getReceiverAccountNumber());
        }

        // 2. 집합(root) 기준으로 파티셔닝
        Map<String, List<ReserveOnceTransferSchedule>> partitions = new HashMap<>();
        for (ReserveOnceTransferSchedule schedule : schedules) {
            String root = find(parent, schedule.getSenderAccountNumber());
            partitions.computeIfAbsent(root, k -> new ArrayList<>()).add(schedule);
        }

        // 3. 파티션별 병렬 실행, 파티션 내부 순차 처리
        List<CompletableFuture<Void>> futures = partitions.values().stream()
                .map(partition -> CompletableFuture.runAsync(() -> {
                    for (ReserveOnceTransferSchedule schedule : partition) {
                        try {
                            reserveTransferExecutionService.executeReserveOnceTransfer(schedule);
                        } catch (Exception e) {
                            log.error("예약 이체 실행 실패, 다음 사이클에 재시도 - scheduleId: {}",
                                    schedule.getId(), e);
                        }
                    }
                }, asyncReserveTransferExecutor))
                .toList();

        CompletableFuture.allOf(futures.toArray(CompletableFuture[]::new)).join();
    }

    private String find(Map<String, String> parent, String x) {
        if (!parent.containsKey(x)) {
            parent.put(x, x);
        }
        if (!parent.get(x).equals(x)) {
            parent.put(x, find(parent, parent.get(x)));
        }
        return parent.get(x);
    }

    private void union(Map<String, String> parent, String a, String b) {
        String rootA = find(parent, a);
        String rootB = find(parent, b);
        if (!rootA.equals(rootB)) {
            parent.put(rootA, rootB);
        }
    }
}
```

## 9. CompletableFuture 병렬 처리 동작 방식

### 구조: 큐를 활용한 Dispatcher-Worker 패턴

- **Dispatcher**: `runAsync`를 호출하는 스케줄러 스레드 — task를 큐에 제출
- **Queue**: `ThreadPoolTaskExecutor` 내부의 `BlockingQueue` (capacity: 10,000) — task 대기열
- **Worker**: 스레드 풀의 워커 스레드 8개 — 큐에서 task를 꺼내 실행

`CompletableFuture`는 이 패턴 위에 **future(결과 추적/조합)** 기능을 얹은 것. `allOf().join()`으로 전체 완료 대기, `thenApply()` 등으로 결과 체이닝이 가능한 점이 단순 dispatcher-worker와의 차이.

### 동작 흐름 (파티션 100개, 워커 8개 예시)

```
작업 큐
┌──────────────────────────────────────────────────┐
│ 파티션1 │ 파티션2 │ 파티션3 │ ... │ 파티션100      │
└──────────────────────────────────────────────────┘

워커1: [파티션1 처리...완료] → [파티션9...완료] → [파티션17...]
워커2: [파티션2 처리...완료] → [파티션10...완료] → [파티션18...]
워커3: [파티션3 처리.........완료] → [파티션11...]  ← 큰 파티션이라 오래 걸림
...
워커8: [파티션8 처리...완료] → [파티션16...완료] → [파티션24...]
```

빨리 끝난 워커가 큐에서 다음 파티션을 가져가므로, 자연스러운 부하 분산이 이루어짐.

### `allOf().join()`의 역할

- 스케줄러 스레드가 모든 파티션 처리 완료까지 **블로킹**
- Spring `@Scheduled`는 기본 단일 스레드이므로, 이전 실행이 끝나야 다음 cron 트리거 실행 가능
- 이를 통해 **중복 실행 방지** 역할도 수행

### `allOf().join()`이 없다면?

- 스케줄러 스레드: task를 큐에 제출만 하고 **바로 메서드를 빠져나옴**
- 워커 스레드: 영향 없음, 큐에 들어간 task는 계속 처리됨
- **문제**: 다음 cron 사이클에 같은 미처리 건을 다시 조회 → 중복 제출 → 이체 중복 실행 위험

### 처리가 cron 주기를 초과하면?

```
00:00  스케줄러 실행 → join()에서 대기 중... (1분 30초 걸리는 작업)
00:30  cron 트리거 발생 → 스레드가 점유 중 → 스킵
01:00  cron 트리거 발생 → 아직 점유 중 → 스킵
01:30  join() 완료 → 메서드 리턴 → 스레드 해제
02:00  cron 트리거 발생 → 스레드 비어있음 → 실행
```

밀린 사이클이 쌓이지 않고, 건너뛰고 다음 cron 시점에 실행됨.

## 10. 트랜잭션 범위와 락 점유 시간 논의

### 현재 트랜잭션 범위

```
LOCK 획득 ─────────────────────────────────────────── LOCK 해제
  │ 계좌 조회 │ 검증 │ 잔액 갱신 │ 거래결과 INSERT │ 원장 INSERT │ COMMIT
  │◄── 핵심 ──────────────────►│◄── 부가 기록 ──►│
```

거래결과, 원장 INSERT는 계좌 row lock과 무관한 작업인데, 같은 트랜잭션에 묶여 있어서 락 해제가 지연됨. 파티셔닝으로 파티션 간 충돌은 없어졌지만, 파티션 내 순차 처리 속도 자체가 느려지는 원인.

### 원장(회계) 저장을 비동기로 분리하면?

#### 가능은 하지만 리스크가 있다

```
트랜잭션 (락 점유 최소화):
  계좌 조회(LOCK) → 검증 → 잔액 갱신 → 거래결과 저장 → COMMIT(LOCK 해제)

비동기:
  원장 기록
```

- **장점**: 락 점유 시간이 짧아져 처리량 증가
- **리스크**: 이체는 성공했는데 원장 기록이 실패하면 **회계 불일치** 발생. 은행 도메인에서 심각한 문제.

#### 실무에서 사용하는 패턴

"그냥 비동기"가 아니라 **반드시 기록된다는 보장 장치**를 함께 사용:

| 패턴 | 방식 | 보장 수준 |
|------|------|----------|
| **Transactional Outbox** | 핵심 트랜잭션에서 outbox 테이블에 이벤트 INSERT → 별도 프로세서가 읽어서 원장 기록 | 강함 (같은 트랜잭션) |
| **@TransactionalEventListener** | 커밋 후 이벤트 발행 → 리스너에서 원장 기록 | 중간 (앱이 죽으면 유실) |
| **CDC (Change Data Capture)** | DB 변경 로그를 감지하여 후속 처리 | 강함 (DB 레벨) |

이 프로젝트에는 이미 `AccountingValidationScheduler`가 매일 4시에 회계 검증을 돌리고 있으므로, 비동기 기록이 실패해도 다음날 검증에서 잡힐 수 있는 구조이긴 함.

### Transactional Outbox 패턴 동작 방식

무거운 회계 기록 대신 **가벼운 이벤트 정보만 핵심 트랜잭션에 저장**하고, 별도 프로세서가 비동기로 조회하여 실제 회계 기록을 수행:

```
트랜잭션 (락 점유 최소화):
  계좌 조회(LOCK) → 잔액 갱신 → 거래결과 저장 → outbox INSERT (가벼움) → COMMIT

별도 프로세서 (비동기):
  outbox 테이블 폴링 → 원장 기록 (무거움) → 처리 완료된 outbox row 삭제/마킹
```

outbox INSERT가 핵심 트랜잭션과 **같은 트랜잭션**에 포함되므로 "이체는 성공했는데 이벤트가 유실되는" 상황이 없음. 커밋되면 둘 다 저장, 롤백되면 둘 다 취소.

## 11. Outbox 패턴 적용 시 처리량 변화 추정

### 건당 처리 비용 비교

**현재 (원장 기록 포함)**:
```
LOCK 획득 ──────────────────────────────────────────────── LOCK 해제
  SELECT FOR UPDATE ×2   ~4ms
  잔액 UPDATE             ~2ms
  거래결과 INSERT ×2       ~3ms
  원장 INSERT (복식부기)    ~3-4ms   ← 이 부분이 제거됨
  스케줄 DELETE            ~2ms
                    합계: ~14-15ms/건
```

**Outbox 적용 시**:
```
LOCK 획득 ──────────────────────────────────── LOCK 해제
  SELECT FOR UPDATE ×2   ~4ms
  잔액 UPDATE             ~2ms
  거래결과 INSERT ×2       ~3ms
  outbox INSERT ×1        ~1.5ms   ← 단순 row 1개
  스케줄 DELETE            ~2ms
                    합계: ~12.5ms/건
```

**건당 약 2-2.5ms 절약, 락 점유 시간 ~15-17% 감소**

### 1만 건 시나리오별 비교

| 시나리오 | 현재 | Outbox 적용 |
|----------|------|-------------|
| 건당 락 점유 | ~15ms | ~12.5ms |
| 고른 분포 (1,000 파티션, 파티션당 10건) | ~19초 | ~16초 |
| 허브 계좌 (5,000건 단일 파티션) | ~75초 | ~63초 |

극적인 차이는 아님. 원장 INSERT가 전체 트랜잭션에서 차지하는 비중이 크지 않기 때문. Outbox 패턴의 진짜 가치는 처리량 향상보다 **원장 기록 실패가 이체 트랜잭션을 롤백시키지 않는 안정성**에 있음.

## 13. 남아있는 한계 및 향후 고려사항

- **파티션 편향(skew)**: 특정 계좌(공과금 수납 계좌 등)가 많은 스케줄에 등장하면 대부분의 건이 하나의 파티션에 몰려 사실상 순차 처리가 됨
- **이중 잠금 구조**: 비관적 락 + bulk UPDATE가 공존. 같은 트랜잭션 내에서 이미 획득한 락에 재접근 시 대기는 없으나, 불필요한 DB 라운드트립 1회 + 영속성 컨텍스트 불일치 문제가 있음. 엔티티 직접 수정(`entity.setBalance()`) + dirty checking 방식으로 전환하면 bulk UPDATE를 제거할 수 있음
- **Outbox 패턴 적용**: 원장 기록을 비동기로 분리하면 락 점유 시간 ~15-17% 감소 + 원장 기록 실패로 인한 이체 롤백 방지. 단, 별도 프로세서와 outbox 테이블이 필요
- **낙관적 락 전환 검토**: 충돌 빈도가 낮다면 `@Version` 기반 낙관적 락이 더 높은 처리량을 제공할 수 있음
